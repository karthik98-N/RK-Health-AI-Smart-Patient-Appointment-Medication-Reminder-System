# RK Health – Application Architecture & System Design

This document details the software architecture, data flow, and functional specifications of the **RK Health – AI Smart Patient Appointment & Medication Reminder System**. It is structured for integration into technical project reports.

---

## 1. Architectural Diagram

The application implements a decoupled, client-server model capable of running in two deployment architectures:
1. **Hybrid Local/Cloud Stack**: Client UI running alongside a local Python Flask web service and SQLite database.
2. **Serverless Cloud Stack**: Client UI hosted on GitHub Pages connecting directly to Google Sheets database via a Google Apps Script Web App.

## 2. Frontend Functionality

The frontend interface is a responsive, single-page application (SPA) built using HTML5, CSS3, and Vanilla JavaScript. It is organized into distinct logical modules:

*   **Lock Screen / Authorization Entry**: Restricts access until the session is authenticated.
*   **Navigation & Sidebar Management**: Smooth transition between dashboard sections (Dashboard, Appointments, Medications, Patient Directory, Reports). Fully responsive, utilizing dynamic drawer overlays for mobile viewports.
*   **Medical Appointment Form**: Receives patient demographics, symptom logs, department routing, priority level, and scheduling details. Includes real-time validation checks.
*   **Medication Scheduler**: Handles the creation of drug reminders, including dose parameters, schedules (e.g., Morning, Night), and patient contact numbers.
*   **Dynamic Health Logs**: Live-updating tables and grid displays presenting patients, scheduled appointments, and medicine trackers.
*   **AI Summary Modal**: Interactive modal displaying patient summaries generated by the LLM. Displays dynamic loading spinners while the backend generates content.
*   **Printable Reports Generator**: Compiles demographic info, compliance statistics, appointment records, and AI summaries into a structured, print-ready document.

## 3. Backend Responsibilities & API Functions

The backend serves as the API gateway, managing data mutations, communicating with external cloud microservices, and parsing database schemas.

### Core Google Apps Script Backend Functions (`code.gs`)

*   **`doGet(e)`**: Receives incoming GET requests containing action query parameters (e.g., `action=patients`). Translates the request into table queries and returns the matching sheet rows as JSON with proper CORS headers.
*   **`doPost(e)`**: Parses incoming POST payloads containing JSON commands (e.g., `addAppointment`). Resolves appropriate CRUD sub-routines.
*   **`addLog(data)`**: Inserts new patient, medication, or appointment logs into their respective sheet tables, dynamically generating incremental IDs.
*   **`getLogs(sheetName)`**: Reads tabular ranges, maps column headers to JSON keys, and structures data arrays.
*   **`updateLog(id, updateData)`**: Locates targeted rows by ID and performs atomic updates (such as updating compliance values or marking medications as "Completed").
*   **`deleteLog(id, sheetName)`**: Locates the target record ID and deletes the corresponding row.
*   **`getStats()`**: Calculates metrics including overall compliance rates, active medication count, and pending appointments.
*   **`generateSummary(patientData)`**: Orchestrates structured requests to the AI engine.
*   **`sendSMS(payload)`**: Dispatches outbound SMS messages to patient numbers.
*   **`runOnce()`**: Automatically seeds initial mock data and headers into empty Google Sheet tabs to guarantee correct first-run execution.

## 4. AI Integration Points

The application connects to the **Groq LLM API** utilizing the **`llama-3.3-70b-versatile`** model to translate clinical details into easy-to-understand instructions.

### The AI Pipeline Workflow:

```
[UI Visit Notes Form] 
       │
       ▼
[JSON Payload: Name, Notes, Meds] 
       │
       ▼
[ai_service.py System & User Prompts]
       │
       ▼ (Groq API: llama-3.3-70b-versatile | Temp: 0.8 | max_tokens: 2048)
[Groq Cloud LLM Processing]
       │
       ▼ (Returns Structured JSON Output)
[Validation & Logging (Success Metrics)]
       │
       ▼
[Update Database & Render UI Report]
```

### Prompt Engineering and Output Validation:
*   **System Prompt**: Enforces the role of a professional medical assistant, instructing it to translate complex clinical terminology into patient-friendly language while restricting the output format strictly to structured JSON.
*   **User Payload**: Consolidates patient name, symptoms, doctor notes, and current prescriptions into a single payload.
*   **Enforced JSON Response Structure**:
    ```json
    {
      "visit_overview": "Sentence overview summarizing the visit outcomes.",
      "diagnosis_explanation": "Simplified description of the medical findings.",
      "medication_instructions": "Step-by-step medication management guidelines.",
      "follow_up_advice": "Recommended timeline and lifestyle actions."
    }
    ```
*   **Fallback Mocks**: In case of rate limits, internet faults, or missing credentials, the service triggers a mock summary generator, preventing application crashes.
